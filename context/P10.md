# Power of Ten

This document is a concise rewrite of *The Power of Ten: Rules for Developing Safety Critical Code* by Gerard J. Holzmann. 

## Why this exists

Safety-critical code has to be predictable. That means:

- control flow should be easy to follow
- memory use should be bounded and understandable
- failures should be detected early
- tools should be able to verify as much as possible

These rules intentionally trade some flexibility for clarity and analyzability. That trade is usually worth it in embedded, flight, robotics, infrastructure, and other high-consequence systems.

## The ten rules

### 1. Keep control flow simple
Do not use:

- `goto`
- `setjmp` / `longjmp`
- recursion
- indirect control flow that hides what runs next

**Why:** simple control flow is easier to read, easier to test, and much easier for static analysis tools to reason about. Banning recursion also helps prove that call graphs are acyclic and bounded.

**Contributor note:** early returns for errors are fine when they make code simpler.

---

### 2. Every loop must have a clear upper bound
Each loop must have a statically understandable maximum number of iterations.

**Why:** bounded loops help prevent runaway behavior and make worst-case execution easier to reason about. If a loop is meant to run forever, that should also be explicit and provable.

**Contributor note:** if traversal length depends on runtime data, add an explicit limit and fail safely if it is exceeded.

---

### 3. No dynamic allocation after initialization
Do not allocate or free heap memory during normal runtime.

**Why:** dynamic allocation introduces fragmentation, timing unpredictability, allocation failure paths, use-after-free bugs, leaks, and harder-to-prove memory behavior. Fixed memory use is easier to validate.

**Contributor note:** allocate what you need up front. Prefer static storage, fixed-capacity containers, or stack allocation with known bounds.

---

### 4. Keep functions small
A function should fit on one printed page. As a practical rule, aim for **about 60 lines or fewer**.

**Why:** a function should be a single understandable unit. Long functions usually hide too many responsibilities and are harder to verify.

**Contributor note:** split large functions by responsibility, not just by line count.

---

### 5. Use assertions to protect assumptions
Each function should contain meaningful assertions. Assertions must:

- check real assumptions
- have no side effects
- fail loudly
- lead to explicit recovery or error propagation

**Why:** assertions catch impossible or abnormal states close to where they happen. They strengthen defensive programming and make intended invariants visible. Empty or trivial assertions do not count.

**Contributor note:** good assertion targets include parameters, state transitions, loop invariants, bounds, and return values from lower-level operations.

---

### 6. Minimize scope
Declare variables at the smallest scope that makes sense.

**Why:** tighter scope reduces accidental misuse, limits where corruption can occur, and makes debugging easier because there are fewer places a value could have changed.

**Contributor note:** do not reuse one variable for multiple unrelated jobs.

---

### 7. Check inputs and return values
- Every function must validate its parameters.
- Every caller must check the return value of non-`void` functions.

**Why:** unchecked failures silently spread upward and become harder to diagnose later. Safety-critical code should make error handling explicit. Ignoring a return value should be rare and intentional.

**Contributor note:** if a return value is intentionally ignored, make that obvious, such as by an explicit cast to `void` and a comment when needed.

---

### 8. Keep the preprocessor simple
Use the preprocessor only for:

- header inclusion
- simple macros
- include guards or similarly limited build guards

Avoid:

- token pasting
- variadic macro tricks
- recursive macros
- heavy conditional compilation

**Why:** preprocessor complexity hides real code structure, confuses tools, and can multiply the number of code variants that must be tested.

**Contributor note:** if a macro is necessary, it should expand into a clean, obvious syntactic unit.

---

### 9. Restrict pointer usage
Prefer direct objects and references to deep pointer structures. Avoid:

- more than one level of dereferencing
- hiding pointer behavior in macros or typedefs
- function pointers unless they are strongly justified

**Why:** pointers make data flow and control flow harder to inspect, reason about, and verify. Function pointers are especially expensive for static analysis.

**Contributor note:** use pointer-based designs only when they are clearly needed and their behavior remains transparent.

---

### 10. Treat warnings as defects
Compile from day one with the strictest warning settings available. Run static analysis regularly. The standard is **zero warnings**.

**Why:** modern compilers and analyzers catch real issues early. Even when a warning appears questionable, the safest response is usually to rewrite the code so the intent becomes obvious to both humans and tools.

**Contributor note:** do not normalize warning debt. Fix it immediately.

## What this means in practice

When writing or reviewing code in this project, prefer code that is:

- bounded rather than open-ended
- explicit rather than clever
- local rather than globally entangled
- checked rather than assumed
- analyzable rather than merely functional

A solution that is slightly more verbose but obviously correct is usually better than a shorter solution that depends on subtle behavior.

## Guidance for AI-generated code

When generating or editing code for this project, the model should:

1. avoid recursion, dynamic allocation, and hidden control flow
2. give loops explicit bounds whenever possible
3. keep functions focused and small
4. add meaningful assertions and error checks
5. validate parameters before use
6. check and propagate failure paths
7. minimize variable scope
8. avoid macro cleverness
9. avoid deep pointer indirection and function pointers unless necessary
10. prefer code that compiles cleanly under strict warnings and is friendly to static analysis

The model should not optimize for cleverness, terseness, or abstraction depth when those make behavior harder to verify.

## Reviewer checklist

Use this as a quick pass during review:

- Is the control flow straightforward?
- Does every loop have a clear bound?
- Is runtime memory behavior fixed and understandable?
- Is each function small and focused?
- Are assumptions enforced with real assertions?
- Are variables scoped as tightly as possible?
- Are inputs and return values checked?
- Is the preprocessor usage minimal and clear?
- Is pointer usage simple and justified?
- Does the code build and analyze with zero warnings?

## Final takeaway

The core idea is simple: **small rule sets work when they are strict, memorable, and enforceable**. These ten rules are not about style for style’s sake. They exist to make critical code easier to trust. That is the real goal.
